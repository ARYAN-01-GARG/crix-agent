# crix — Context-Routed Index eXecutor

An enterprise-grade CLI coding agent that routes tasks to specialist AI agents using an indexed map of your codebase — so every agent gets exactly the context it needs, nothing more.

```
$ crix "add JWT authentication to the user routes"

  [orchestrator] classified: medium task → backend specialist
  [router] relevant files: src/routes/user.ts, src/middleware/, src/types/
  [backend] reading 3 files...
  [backend] writing src/middleware/auth.ts
  [backend] updating src/routes/user.ts
  [backend] done — 2 files changed, 87 insertions

  Task complete. Run `crix done` to update the project index.
```

---

## What makes crix different

Most coding agents read the entire codebase to answer every question. crix builds and maintains a **hybrid AST + semantic index** of your project, then routes each task to the right specialist with only the relevant slice of that index. Small changes stay cheap. Large changes scale cleanly.

| | Generic coding agents | crix |
|---|---|---|
| Context for a 1-file fix | Entire codebase | That file's index entry |
| Agent model | One general agent | Orchestrator + domain specialists |
| Token cost on repeat calls | Full price every time | ~10% via prompt caching |
| Infrastructure | Local only | Local CLI or self-hosted server with queuing |

## Core concepts

### The project index (`.crix/`)

When you run `crix init`, crix scans your project and creates `.crix/` at the root with four knowledge files:

| File | Purpose | Default size |
|---|---|---|
| `project.md` | Project overview, goals, tech stack | 2,500 chars |
| `tree-structure.md` | Every file → all functions/classes with summaries | 8,000 chars |
| `context.md` | Current working context, recent decisions | 2,500 chars |
| `code-practices.md` | Coding standards, patterns, anti-patterns | 2,500 chars |

`tree-structure.md` is generated by combining two layers:
- **AST layer** (tree-sitter) — extracts exact function names, signatures, and types. Zero LLM cost. Always accurate.
- **Semantic layer** (LLM, batched per file) — generates 1–2 line summaries of what each function does.

```
## src/auth/jwt.ts
signToken(payload: JwtPayload, secret: string): string
  → Signs a JWT with HS256; returns a signed token string
verifyToken(token: string, secret: string): JwtPayload
  → Verifies token signature and expiry; throws on invalid or expired
refreshToken(oldToken: string, secret: string): string
  → Issues a new token if the old token is within the refresh window
```

Index files update only when you confirm a task is complete — not on every save. This keeps LLM costs bounded.

### Orchestrator → Specialist model

crix never uses a single monolithic agent. Every request goes through the **orchestrator**, which:

1. Classifies the task (tiny / small / medium / large) using heuristics first, LLM only if ambiguous
2. Identifies which files are in scope by reading the index
3. Routes to the right specialist with only the relevant index slices

```
Orchestrator (Haiku — cheap routing)
  ├─ backend    — API routes, databases, auth, services
  ├─ frontend   — UI components, state, styling
  ├─ tester     — unit tests, integration tests, mocks
  ├─ reviewer   — code review, security checks
  ├─ devops     — CI/CD, Docker, infrastructure
  └─ designer   — UI/UX, design systems
```

Specialists are flat — only the orchestrator assigns work. This keeps execution traceable and cost predictable.

### Modes

```
crix mode plan    # Agents read and propose — no file writes
crix mode work    # Full tool access — reads, writes, shell
crix mode review  # Agents read and suggest — no file writes
```

### Sessions

Sessions live in `.crix/sessions/` in your project. Conversation history, mode state, and task results persist across invocations.

```
crix session list
crix session resume <id>
```

## Installation

> Requires Node.js 20+ and pnpm.

```bash
# Install globally
npm install -g crix

# Or run without installing
npx crix init
```

### Authenticate

```bash
crix login
```

This opens your browser to the crix authentication page. Sign in with your credentials — the CLI automatically receives your session token in the background. No copy-pasting keys.

```
$ crix login

  Opening browser for authentication...
  If it didn't open → https://auth.crix.dev/login?state=x9k2m&port=54871

  ⠋ Waiting for authentication...

  ✓  Logged in as aryan@example.com
     Token stored in system keychain.
```

If you're on SSH or a headless machine, crix falls back to the **device code flow** automatically:

```
$ crix login

  Cannot open browser — headless environment detected.

  Visit  : https://auth.crix.dev/device
  Code   : XKCD-4829

  ⠋ Waiting... (times out in 5 min)
  ✓  Logged in as aryan@example.com
```

```bash
# Sign out and revoke the token server-side
crix logout

# Show currently authenticated user
crix whoami
```

### Initialize a project

```bash
cd your-project
crix init
```

This scans your project, builds the index, and creates `.crix/`. Add `.crix/sessions/` to `.gitignore`; commit the four knowledge files to share context with your team.

## Usage

```bash
# Authenticate
crix login            # opens browser, stores token in system keychain
crix logout           # revokes token server-side and clears keychain
crix whoami           # show current user

# Start a session
crix start

# One-shot task
crix "refactor the auth middleware to use async/await"

# Specify mode inline
crix plan "design the database schema for user preferences"
crix work "implement the schema you just planned"
crix review "check the changes I just made"

# Update context files after completing a feature
crix done

# Update context files manually
crix context update

# Manage sessions
crix session list
crix session resume abc123

# Show current index
crix index show
crix index show src/auth/
```

## Configuration

Create `crix.config.ts` in your project root (or a parent directory):

```ts
export default {
  // LLM provider: "anthropic" | "openai" | "google" | "ollama"
  provider: "anthropic",
  model: "claude-sonnet-4-6",
  apiKey: process.env.ANTHROPIC_API_KEY,

  // Adjust context file size limits (in characters)
  contextLimits: {
    projectMd: 2500,
    treeStructure: 8000,
    contextMd: 2500,
    codePractices: 2500,
  },

  // Choose which specialists are available
  agents: {
    enabled: ["backend", "frontend", "tester", "reviewer", "devops"],
  },

  // Harness permissions — control what agents can do
  harness: {
    permissions: [
      { tool: "run-shell", pattern: "rm -rf*", level: "deny" },
      { tool: "write-file", pattern: ".env*", level: "ask" },
    ],
  },

  // Optional: tiered model config
  models: {
    router: "claude-haiku-4-5",   // cheap model for routing decisions
    worker: "claude-sonnet-4-6",  // capable model for actual work
  },
}
```

## Authentication

crix uses the **OAuth localhost callback flow** — the same pattern as GitHub CLI, Vercel CLI, and Google Cloud SDK. Credentials are entered in a browser on crix's domain (HTTPS). The CLI never handles your password.

### How it works

```
CLI                              Browser                        crix server
 │                                  │                               │
 │  1. generate state + PKCE        │                               │
 │  2. start local server :54871    │                               │
 │  3. open browser ──────────────► │                               │
 │                                  │  4. user enters credentials   │
 │                                  │ ─────────────────────────────►│
 │                                  │                               │  5. validate
 │                                  │◄─────────────────────────────-│  6. redirect to
 │                                  │     localhost:54871/callback      localhost
 │◄─────────────────────────────────│         ?token=xxx
 │  7. capture token
 │  8. save to keychain
 │  9. shut down local server
```

**Security properties:**
- PKCE (Proof Key for Code Exchange) prevents auth code interception
- `state` parameter prevents CSRF attacks
- Token stored in OS keychain — never written to disk as plaintext
- Auth codes are single-use and short-lived (60s)
- Local server only listens on `127.0.0.1` (not exposed to network)

### Device code fallback

In headless or SSH environments where a browser cannot be opened, crix automatically switches to the device code flow:

```
CLI polls GET /auth/device/poll every 5s
  → pending: keep waiting
  → authorized: exchange device code for token
  → expired: prompt user to restart
```

### Token storage

| OS | Storage |
|---|---|
| macOS | Keychain via `keytar` |
| Linux | Secret Service (libsecret) via `keytar` |
| Windows | Credential Manager via `keytar` |
| Fallback | `~/.crix/credentials` (AES-256 encrypted, key derived from machine ID) |

## Architecture

```
packages/
  shared/        Types, logger, errors (base dependency)
  auth/          Login flow — localhost callback server, device code fallback,
                 PKCE, browser opener, system keychain token storage (keytar)
  core/          Sessions (SQLite), mode state machine, config loader
  harness/       Tool execution layer — all file/shell/git ops, permissions, hooks
  context/       Manages the 4 .crix/ knowledge files; slices and budgets context
  indexer/       tree-sitter AST + LLM summarizer → tree-structure.md
  agents/        BaseAgent + specialist implementations
  orchestrator/  Task classifier, router, specialist dispatcher, result merger
  cli/           `crix` binary, Ink TUI, all commands
  queue/         BullMQ job definitions (server mode)
  cache/         Redis client + caching utilities (server mode)
  events/        EventEmitter abstraction (local: Node.js / server: Redis Streams)

apps/
  server/        Fastify server — REST API + SSE streaming + BullMQ workers
                 Auth endpoints: /auth/login, /auth/callback, /auth/device,
                                 /auth/device/poll, /auth/logout, /auth/me
  worker/        Standalone BullMQ worker process (scale independently)
```

### Token efficiency

- **Task classification** — tiny tasks skip the routing LLM call entirely; heuristics handle them
- **Context slicing** — specialists receive only the relevant sections of the index, not the full file
- **Prompt caching** — system prompt + project.md + code-practices.md are cached; repeat calls cost ~10% of original
- **Tiered models** — cheap model (Haiku) for routing; capable model (Sonnet) for coding work
- **History compression** — older session turns are summarized, not sent verbatim

### Harness layer

Every tool call an agent makes flows through the harness:

```
agent tool call
  → permissions.check()     deny → HarnessPermissionError
  → hooks.pre()             user-defined pre-hooks
  → tool.execute()          actual operation
  → hooks.post()            user-defined post-hooks (e.g. run formatter)
  → return to agent
```

### Server mode (team / enterprise)

Running `crix-server` alongside a Redis instance enables remote access, team sessions, and full audit logging:

```
Client → POST /api/tasks
       → BullMQ queue (Redis-backed)
       → Agent worker executes task
       → Events emit to Redis Streams
Client ← GET /api/tasks/:id/stream (SSE)
       ← Live streamed output
```

The event system is pluggable: swap Redis Streams for Kafka in high-throughput enterprise deployments without changing application code.

## Development

### Prerequisites

- Node.js 20+
- pnpm 9+
- Redis (for server mode — Docker Compose provided)

### Setup

```bash
git clone https://github.com/your-org/crix-agent
cd crix-agent
pnpm install
pnpm build
```

### Run tests

```bash
pnpm test
```

### Lint and format

```bash
pnpm lint        # Biome lint
pnpm format      # Biome format
```

### Local development

```bash
# Build and watch all packages
pnpm dev

# Run the CLI from source
node packages/cli/dist/index.js init

# Start Redis for server mode
docker compose up redis -d

# Start the server
pnpm --filter @crix/server dev
```

### Git branches

```
main         Protected — tagged releases only
develop      Integration branch — PRs target here
feature/xxx  New features
fix/xxx      Bug fixes
```

Commits follow [Conventional Commits](https://www.conventionalcommits.org/): `feat:`, `fix:`, `chore:`, `docs:`, `test:`.
